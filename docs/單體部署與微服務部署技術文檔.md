# Rocket.Chat 部署架構：從單體式到微服務

## 前言：開發與生產的差異

在本地端進行開發時，我們通常透過 `yarn dev` 等指令以**單體式 (Monolithic)** 模式運行 Rocket.Chat。此模式將所有服務整合在單一應用程式中，旨在簡化開發流程，讓開發者能快速啟動與測試。

然而，對於需要應對大量使用者和高併發的**生產環境 (Production)**，官方強烈建議採用**微服務 (Microservices)** 架構來部署，以實現更佳的擴展性、穩定性和資源利用率。

## 核心概念：部署方式的轉變

從單體式轉向微服務，關鍵在於**改變部署方式**，而非重寫程式碼。Rocket.Chat 的程式碼庫本身已高度模組化（如 `packages/` 目錄所示），微服務部署就是將這些模組作為獨立的服務來運行和擴展。

此過程主要依賴兩項核心技術：

### 1. Docker：服務容器化

Docker 用於將 Rocket.Chat 的不同部分（如 Web 應用、推播通知服務等）打包成獨立、標準化的**容器 (Container)**。每個容器都封裝了運行該服務所需的所有環境和依賴。

### 2. Kubernetes：容器編排

Kubernetes (K8s) 作為容器編排工具，負責全自動地管理大規模的 Docker 容器。其主要職責包括：

-   **自動部署與擴展**：根據負載自動增加或減少服務的實例數量。
-   **服務發現與負載平衡**：讓各服務間能互相溝通，並將流量平均分配到各個實例。
-   **自我修復**：當某個服務實例發生故障時，自動重新啟動一個新的實例。

## Rocket.Chat 微服務架構概覽

一個典型的 Rocket.Chat 微服務部署包含以下組件：

-   **Web App 實例**：核心的 Meteor 應用程式，可運行多個實例來分散使用者請求。
-   **Gateway**：所有微服務的統一入口，負責請求路由。
-   **獨立後端服務**：將核心功能拆分為獨立服務，可獨立擴展，例如：
    -   `Presence`：管理使用者在線狀態。
    -   `Stream Hub`：處理即時數據流。
    -   `DDP Streamer`：處理 DDP (Distributed Data Protocol) 通訊。
    -   `Notification`：處理推播通知。
-   **MongoDB 資料庫**：所有服務共享的資料庫。在生產環境中，應使用高可用的 Replica Set 叢集。
-   **訊息中間件 (Transporter)**：通常使用 `NATS` 或 `Redis`，讓各微服務之間可以高效地進行非同步通訊和數據交換。

## 如何進行微服務部署？

標準的部署流程如下：

1.  **環境準備**：準備一個 Kubernetes 叢集（可使用 GKE, EKS, AKS 等雲端服務或自建）。
2.  **映像檔建置**：使用專案中提供的 `Dockerfile` 為 Rocket.Chat 應用程式建置 Docker Image。
3.  **設定部署文件**：Rocket.Chat 官方提供 **Helm Chart** 作為 K8s 的標準部署方案。開發者需根據自身環境（如資料庫位址、網域名稱等）修改其 `values.yaml` 設定檔。
4.  **部署**：使用 `helm` 指令將應用程式部署到 K8s 叢集。Helm 會自動建立所有必要的資源。
5.  **擴展 (Scaling)**：部署後，可使用 `kubectl scale` 指令根據需求獨立擴展某個服務。例如，僅增加 `notification` 服務的實例數量以應對高流量。

## 總結與建議

-   **本地開發**：繼續使用**單體式**模式，以求開發效率。
-   **生產部署**：
    -   **首選方案**：使用 **Kubernetes** 搭配官方的 **Helm Chart**。這是功能最完整、擴展性最強的方案。
    -   **簡化方案**：對於小規模部署，可參考 `docker-compose-*.yml` 文件，使用 **Docker Compose** 啟動多容器環境。此方案易於管理，但擴展性不如 K8s。
